# SmartThings Gateway 공개 출시 계획

기준일: 2026-07-22

이 문서는 현재 구현과 공식 요구사항 사이의 차이를 추적하는 엔지니어링 출시
게이트입니다. 법률 자문이나 SmartThings의 승인을 대신하지 않습니다. 확인되지 않은
운영자 정보, 상업 조건, 인프라 사업자, 데이터 처리 지역은 코드나 공개 문구에서
추정하지 않습니다.

## 현재 결론

Gateway는 비공개 기술 MVP로 검증할 수 있지만, 불특정 사용자가 가입하는 공개 서비스나
유료 서비스로 출시할 준비는 아직 끝나지 않았습니다.

가장 먼저 해결해야 하는 외부 게이트는 SmartThings Developer Terms의 다음 두 조항 사이에서
Growful 모델이 허용되는지 서면으로 확인하는 일입니다.

- `Authorized Purpose`는 Licensee Product를 SmartThings 사용자가 이용하도록 운영하는 목적을
  포함합니다.
- 제한 조항은 Developer Tools를 service bureau 또는 Licensee가 아닌 다른 사람의 이익을 위해
  사용하는 것을 금지합니다.

사용자가 발급한 OAuth 토큰을 Growful이 보관하고 자체 Bearer 토큰으로 SmartThings API 요청을
대행하는 모델이 어느 범주에 해당하는지는 문언만으로 확정하지 않습니다. 공개 가입, 유료화,
SmartThings 명칭을 이용한 홍보 전에 SmartThings의 서면 답변 또는 별도 계약을 확보해야 합니다.

공식 근거:

- [SmartThings Developer Terms of Service](https://developer.smartthings.com/termsofservice)
- [SmartThings API Access App Setup](https://developer.smartthings.com/docs/service-integrations/app-setup)
- [SmartThings Product Policies](https://developer.smartthings.com/docs/certification/product-policies)
- [개인정보 보호법 제30조](https://law.go.kr/LSW/lsLinkCommonInfo.do?chrClsCd=010202&lsJoLnkSeq=1020398583)
- [개인정보 보호법 시행령 제31조](https://www.law.go.kr/LSW/lsLawLinkInfo.do?chrClsCd=010202&lsJoLnkSeq=900079801)
- [개인정보 처리방침 작성지침(2026.4.)](https://www.pipc.go.kr/np/cop/bbs/selectBoardArticle.do?bbsId=BS217&mCode=&nttId=12018)

SmartThings에 제출할 질문과 현재 증빙은
[공개·상업 사용 확인 요청 패킷](./SMARTTHINGS-APPROVAL-REQUEST.md)과
[약관·정책 준수 매트릭스](./SMARTTHINGS-COMPLIANCE-MATRIX.md)에서 관리합니다.
3단계 운영 통제의 실행 초안은 [정보보호 프로그램](./INFORMATION-SECURITY-PROGRAM.md),
[사고 대응 Runbook](./INCIDENT-RESPONSE.md),
[데이터 보존·파기 대장](./DATA-RETENTION.md)에서 관리합니다. 세 문서는 승인·운영·훈련 증빙이
완료되지 않은 초안이며, 문서 존재 자체를 출시 통과로 간주하지 않습니다.

## 확인된 현재 데이터 흐름

| 구간 | 처리 내용 | 현재 보존 |
| --- | --- | --- |
| OAuth 시작 | 요청 scope, 정책 버전, 동의 시각, 비공개 베타 사용자명과 무작위 state의 SHA-256 해시 | 콜백에서 1회 소비; 정상 실행 중 미사용 시 생성 후 최대 15분, 재기동 시 즉시 정리 |
| OAuth 완료 | `installedAppId`, 승인 scope, 토큰 종류와 만료·갱신 시각, 정책 동의 증빙 | 연결 해제, 초대·정책 회수 또는 SmartThings 삭제 lifecycle까지 |
| SmartThings 자격 증명 | access token과 refresh token | AES-256-GCM 암호문으로 PostgreSQL에 저장 |
| Growful 자격 증명 | Growful 토큰 | 원문은 1회 표시, SHA-256 해시와 발급 시각만 저장 |
| API 프록시 | 승인된 SmartThings 응답과 요청 본문 | 애플리케이션 DB에는 저장하지 않음; 메모리에서 전달 |
| 갱신 작업 | 최근 갱신 시각, 임대 식별자, 마지막 오류 | 연결 행과 함께 저장 |
| 감사기록 | 연결 식별자의 SHA-256 가명값, 작업·결과·UTC 시각, 이전 이벤트 해시 | PostgreSQL append-only 테이블, 검증 CLI, 시작·유지보수 주기 전체 체인 검증과 readiness 차단; 보존기간·외부 불변 sink는 미확정 |
| 브라우저 관리 | 사용자가 입력한 Growful 토큰 | 현재 탭 메모리만 사용; 쿠키와 브라우저 저장소 미사용 |

현재 `DELETE /connection`과 SmartThings `DELETE` lifecycle은 기본 연결 행을 삭제합니다.
그러나 백업, PostgreSQL WAL, 호스트 스냅샷, 외부 로그까지 즉시 영구 삭제되는지는 저장소
설정만으로 증명되지 않습니다. 따라서 공개 개인정보 처리방침에는 백업 보존기간과 삭제
절차가 확정되기 전 “즉시 완전 삭제”라고 적지 않습니다.

## 요구사항 차이

| 요구사항 | 현재 상태 | 출시 판정 |
| --- | --- | --- |
| OAuth, 토큰 갱신, 연결별 Growful 토큰 | 구현·자동 테스트 완료 | 통과 |
| 토큰 암호화와 원문 로그 차단 | 구현·자동 테스트 완료 | 통과 |
| 선택형 최소 권한과 SmartThings 설치 해제 삭제 | 구현·자동 테스트 완료 | 통과 |
| 개인정보 판매·분석·광고 목적 사용 | 해당 기능 없음 | 유지 필요 |
| 사용자 데이터 삭제 UI | 기본 DB 행 삭제 구현 | 백업·외부 로그 범위 미확정 |
| 공개 개인정보 처리방침 | 내장 `/privacy`에 확인된 데이터 흐름과 미확정 경계를 구현하고 HTTP·배포 검사로 접근성 검증 | 운영 사실값·법률 검토가 없어 차단 |
| 운영자 신원, 개인정보 담당자, 지원 연락처 | 저장소에 확정값 없음 | 차단 |
| 처리위탁자, 처리 국가, 국외 이전 근거 | 인프라 계약·지역 정보 없음 | 차단 |
| 서면 정보보호 프로그램 | 초안 작성, 승인·운영 증빙 미완료 | 부분 |
| 개인정보 접근을 재구성할 수 있는 감사기록 | 연결 수명주기·인증·토큰 읽기의 가명 hash-chain과 자동 무결성 검증 구현 | 중앙 관리자 신원, 외부 불변 sink·보존·독립 alert·운영 검토 증빙 미구현으로 차단 |
| 24시간 내 SmartThings 사고 통지 절차 | runbook 초안 작성, 실제 접수 채널·훈련 미완료 | 부분 |
| 사용자 문의·권리행사 처리 절차 | 내장 `/support`에 안전한 문의 범위, 셀프서비스와 미확정 운영 경계 구현 | 본인 확인·응답 목표·긴급 단계와 실제 티켓 운영이 없어 차단 |
| 사용자 상태 표면 | `/status`가 현재 readiness와 PostgreSQL 기반 운영자 공지·해결 이력을 표시하고 배포·CI에서 접근성 검증 | 자동 탐지·개별 통지·실제 운영 훈련·SLA·가동률 목표가 없어 부분 |
| 비공개 베타 접근 제한 | 사용자별 hash 기반 Basic 초대·실패 제한, PostgreSQL 발급·조회·회수, 즉시 기존 토큰 회수와 운영자 감사 구현 | 중앙 인증된 운영자 신원·지원 절차 전까지 소수 검증자만 허용 |
| 토큰 대행 모델의 SmartThings 허용 여부 | 서면 확인 없음 | 공개·유료 출시 차단 |
| SmartThings 명칭·표장 사용 범위 | 서면 확인 또는 인증 없음 | 홍보 확대 차단 |
| 변경 사전 통지 요구 적용 범위 | Terms에 30일·45일 조항이 존재 | SmartThings 확인 전 변경 출시 차단 |

## 단계별 목표와 종료 조건

### 0단계 — 기술 MVP 기준선

목표: 한 명의 운영자가 통제된 환경에서 OAuth와 프록시 동작을 검증할 수 있게 합니다.

완료 조건:

- 토큰 원문 비노출, 암호화 저장, 갱신 race 방지, 연결별 인증이 자동 테스트를 통과합니다.
- SmartThings webhook 서명과 app ID를 검증하고 설치 해제 시 연결을 삭제합니다.
- 관리 화면은 Growful 토큰을 영구 저장하지 않습니다.

상태: 구현 완료. 이 단계의 완료는 공개 모집 허가를 의미하지 않습니다.

### 1단계 — 실패 차단형 출시 계약

목표: 설정 누락이나 외부 승인 부재 상태에서 공개 모드로 기동하지 못하게 합니다.

구현 범위:

- `private_beta`와 `public`을 구분하는 명시적 서비스 모드를 추가합니다.
- 운영자명과 지원 연락처를 타입 검증하고 정책 URL은 callback과 같은 origin의 내장 경로로 고정합니다.
- `public` 모드는 SmartThings 서면 확인 참조번호와 확인일이 없으면 시작·배포 preflight를
  실패시킵니다.
- 비공개 베타는 이름만 베타인 공개 가입이 되지 않도록 실제 초대 또는 허용 목록으로
  `/oauth/start`를 보호합니다.

종료 조건:

- 설정 단위 테스트, 배포 preflight 실패/성공 테스트, 실제 HTTP 접근 제어 테스트가 모두
  통과합니다.
- 운영 값은 저장소에 하드코딩하지 않고 배포 환경에서만 주입합니다.

상태: 구현 완료. 비공개 Basic 자격 증명은 사용자별로 분리하며 비밀번호 원문은 설정에
저장하지 않습니다. 초대 목록에서 항목을 제거하고 Gateway를 재기동하면 해당 사용자의 기존
Growful·SmartThings 토큰도 회수합니다. PostgreSQL 관리 CLI는 256-bit secret을 발급 시 한 번만
표시하고, 조회에서는 자격 증명 material을 제외하며, 회수 시 미완료 OAuth state와 기존 연결을
원자적으로 삭제합니다. 설정 초대와 같은 이름의 DB 회수 행은 tombstone으로 우선하며, 발급·회수
작업은 해시된 운영자 ID와 ticket으로 감사 체인에 기록됩니다. 중앙 인증된 운영자 신원과 SSH
session audit의 실제 결합이 완료되기 전까지 소수의 통제된 검증자에게만 발급합니다.

### 2단계 — 개인정보·이용조건·지원 표면

목표: 가입 전에 실제 데이터 흐름과 사용자의 권리를 이해할 수 있게 합니다.

구현 범위:

- 사이트 전체에서 쉽게 찾을 수 있는 정확한 이름의 `개인정보 처리방침`을 제공합니다.
- 처리 목적·항목·보유기간·파기·제3자 제공·위탁·국외 이전·안전조치·권리행사·담당자·변경
  고지를 실제 운영 사실에 맞게 표시합니다.
- SmartThings로의 정보 전달과 이용을 명확히 알리고, OAuth 승인 전에 정책을 볼 수 있게 합니다.
- 이용조건에는 서비스 범위, 사용자 책임, 금지 사용, 가용성, 연결 해제와 SmartThings 설치
  해제의 차이를 설명합니다.
- 지원 페이지에는 문의 채널, 응답 목표, 보안 사고 신고 채널, 데이터 열람·정정·삭제 요청
  절차를 제공합니다.

종료 조건:

- 운영자와 개인정보 담당자가 모든 사실값을 제공하고 법률 검토를 기록합니다.
- SmartThings Product Policies의 개인정보 항목과 한국 개인정보 보호법 제30조·시행령 제31조
  항목을 각각 테스트 가능한 콘텐츠 계약으로 연결합니다.
- 모바일·데스크톱 실제 브라우저에서 키보드, 고대비, 확대, 링크, 폼 흐름을 검증합니다.

상태: 부분 구현. 운영자·지원·정책 링크를 홈과 OAuth 선택 화면에 표시하고 명시적 동의와
정책 버전을 연결에 기록합니다. `/privacy`와 `/terms`는 현재 코드로 확인되는 데이터 흐름,
보관·삭제 경계, 베타 이용 제한을 직접 제공합니다. `/support`는 연결·토큰 노출·개인정보·보안
문의를 비밀값 없이 접수하는 방법과 셀프서비스, 아직 확정되지 않은 본인 확인·응답시간 경계를
설명합니다. `/status`는 Gateway의 현재 readiness와 SmartThings 종단 간 가용성을 보장하지 않는
경계를 설명합니다. 배포와 CI가 네 경로의 `2xx` 응답과 문서 marker를 확인합니다.
운영자 법적 신원, 담당자, 위탁·처리 국가, 백업 보존기간, 공개·상업 조건과 법률 검토는 여전히
미확정입니다. 로컬 브라우저 접근도 조직 정책에 차단되어 모바일·데스크톱 실제 브라우저 검증은
공개 URL 배포 후 다시 수행해야 하므로, 이 문서만으로 공개 출시 조건을 충족하지 않습니다.

### 3단계 — 보안 운영과 삭제 보증

목표: 코드 보안뿐 아니라 운영 중 접근·삭제·사고를 재현하고 대응할 수 있게 합니다.

구현 범위:

- 서면 정보보호 프로그램, 권한 부여·회수, 정기 접근 검토, 비밀 교체, 취약점 대응 절차를
  문서화합니다.
- 개인정보 원문 없이 관리자 접근 주체, 목적, 연결의 가명 식별자, 작업, 결과, 시각을 남기는
  변조 방지 감사기록을 구현합니다.
- DB, WAL, 백업, 호스트 스냅샷, 로그의 보존기간과 삭제 절차를 확정하고 복구 훈련으로
  검증합니다.
- 사고 탐지부터 SmartThings 24시간 통지, 법정 통지, 사용자 안내까지 연습 가능한 runbook을
  만듭니다.

종료 조건:

- 삭제 요청 샘플이 운영 DB에서 사라지고 정해진 백업 수명 뒤 복구할 수 없음을 증명합니다.
- 권한 변경 이력과 개인정보 접근 이력을 독립 검토자가 재구성할 수 있습니다.
- 사고 모의훈련의 시간·담당자·통지 산출물이 남습니다.

상태: 부분 구현. 연결 승인·재승인, Growful 토큰 교체, 연결 삭제, 갱신 성공·실패, 인증과
SmartThings 토큰 읽기를 가명 식별자로 기록합니다. 레코드는 이전 이벤트 해시와 연결되고
애플리케이션 DB 역할의 수정·삭제·truncate를 거부합니다. 운영 CLI와 Gateway가 전체 체인의
형식·순서·이전 해시·이벤트 해시를 재계산합니다. Gateway는 시작 시와 기존 유지보수 주기마다
재검증하고 손상 또는 읽기 실패 동안 `/readyz`를 `503`으로 전환합니다. 그러나 중앙 인증된
관리자 주체, 별도 불변 sink, 보존기간, 검색·독립 alert·운영 검토 증빙은 아직 없습니다.

### 4단계 — 통제된 비공개 베타

목표: 승인된 소수 사용자로 실제 SmartThings 연동과 운영 절차를 검증합니다.

구현 범위:

- 초대 발급·회수, 사용량 제한, abuse 방지, 지원 티켓, 상태 공지를 운영합니다.
- OAuth, 자동 갱신, Growful 토큰 교체, 프록시, 양쪽 연결 해제, 권리행사, 장애 복구를 실사용
  시나리오로 검증합니다.
- 데이터 최소화와 scope 이탈 여부를 베타 결과로 다시 점검합니다.

종료 조건:

- 실제 SmartThings 계정 두 개 이상에서 연결 분리와 삭제가 확인됩니다.
- 지원·삭제·사고 훈련 목표시간을 충족합니다.
- 베타 참여자의 명시적 고지·동의와 피드백 기록이 있습니다.

상태: 부분 구현. SmartThings가 `429`와 `Retry-After`를 반환하면 연결별 backoff 마감을
PostgreSQL에 기록해 모든 Gateway 인스턴스가 같은 대기시간을 준수합니다. 다른 연결은 계속
사용할 수 있습니다. Growful도 인증된 연결별 고정 60초 창에서 60건을 허용하고 61번째부터
PostgreSQL 공유 `429`로 차단합니다. 이 공정 사용 상한은 SmartThings의
[Installed Apps API 분당 60건 기준](https://developer.smartthings.com/docs/getting-started/rate-limits)을
근거로 하며 endpoint별 upstream 제한을 대체하지 않습니다. quota 거부 횟수와 마지막 거부
시각을 연결별로 누적하고, `/connection`에 가명 `supportReference`를 제공합니다. 운영자는 이
참조값으로 반복 위반을 조회해 고정 사유 중 하나로 명시적 차단·해제할 수 있습니다. 차단은
모든 Gateway 인스턴스에 공유되고 `/v1/*`를 upstream 호출 전에 `403`으로 거부하며, 작업은
해시된 운영자 ID·승인 ticket과 함께 감사 체인에 남습니다. 자동 차단 임계값은 SmartThings나
운영 정책 확인 없이 임의로 만들지 않았습니다. 비공개 초대도 PostgreSQL에서 발급·조회·회수할
수 있고 회수 즉시 Basic·Growful 인증을 함께 거부하며 작업을 같은 감사 체인에 남깁니다. 중앙
상태 공지는 운영 CLI로 열기·조사/관찰 갱신·해결할 수 있고 `/status`에 현재 공지와 해결 이력을
표시하며 각 작업을 해시된 운영자 ID·ticket으로 감사 기록합니다. 중앙 인증된 운영자 신원,
자동 anomaly alert, 지원 ticket 연동, 공지 훈련과 실제 SmartThings 계정 두 개 이상의 베타
증빙은 아직 구현되지 않았습니다.

### 5단계 — SmartThings 공개 사용 확인

목표: 공개·상업 모델에 대한 SmartThings 조건을 추정이 아니라 증거로 확정합니다.

확인 질문:

1. OAuth 토큰을 서버에서 보관하고 Growful 토큰으로 API 요청을 대행하는 다중 사용자 서비스가
   Authorized Purpose인지, service bureau 제한에 해당하는지?
2. 무료 공개 베타, 공개 서비스, 유료 서비스 각각에 별도 계약·검토·인증이 필요한지?
3. `SmartThings` 명칭을 서비스 설명과 도메인 페이지에서 사용할 수 있는 범위는 어디까지인지?
4. API Access App에 적용되는 출시 전 테스트, 변경 통지, 개인정보·보안 검토 제출물이 무엇인지?
5. 개인정보 삭제·감사 로그·사고 통지의 구체적 접수 채널과 증빙 형식은 무엇인지?

종료 조건:

- 답변 원문, 담당 조직, 날짜, 적용 서비스 버전이 변경 불가능한 기록으로 보관됩니다.
- 요구된 계약·검토·테스트·사전 통지를 완료합니다.
- 답변과 코드/정책이 충돌하면 공개 모드는 계속 차단합니다.

### 6단계 — 공개 출시와 상업화

목표: 외부 게이트와 내부 검증을 모두 통과한 버전만 공개합니다.

완료 조건:

- 1~5단계의 차단 항목이 모두 증거와 함께 해소됩니다.
- 개인정보 처리방침, 이용조건, 지원, 상태, 삭제 흐름이 공개 도메인에서 작동합니다.
- 공개 모드의 rate limit, abuse 대응, 용량 경보, 비용 한도, 가용성 목표가 검증됩니다.
- 유료화한다면 가격·결제사업자·청약철회/환불·세금·사업자 표시·미성년자 정책을 별도 법률
  검토하고, SmartThings의 상업 이용 허용 범위를 다시 확인합니다.
- 전체 자동 검사, 실제 HTTPS API 시나리오, 실제 브라우저 시각·접근성·성능 QA를 통과합니다.

## 운영자가 제공해야 하는 사실값

다음 값이 확정되기 전에는 최종 정책 문구를 만들거나 공개 모드를 켜지 않습니다.

- 법적 운영자명, 국가, 주소, 사업자 정보(해당 시)
- 대표 지원 연락처와 개인정보 보호 담당자 연락처
- 호스팅·터널·DB·백업·모니터링 사업자와 실제 처리 국가/리전
- 각 저장소와 백업의 보존기간, 삭제 주기, 복구 정책
- 데이터 열람·정정·삭제 요청 처리기한과 신원 확인 방법
- 보안 사고 연락망과 SmartThings 통지 채널
- SmartThings의 공개/상업 사용 서면 답변과 변경 통지 이력
- 무료/유료 여부, 가격, 결제·환불·세금 처리 주체
